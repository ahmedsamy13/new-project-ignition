# Project File Structure Guide (Comprehensive & Detailed)

This document provides a highly detailed, file-by-file breakdown of the React Enterprise Boilerplate. The architecture is based on **Feature-Sliced Design (FSD)**, which prioritizes vertical feature slices over flat file-type directories (e.g., separating all hooks from all components).

Our core philosophy: **Code flows downwards.** A file can only import from files in layers below it. Never above it.

---

## 1. Root Configuration Files (The Engine Room)

These files configure the build tooling, dependencies, testing environments, and CI/CD pipelines. They are the backbone of the project.

- **`.env`**: Local environment variables (e.g., `VITE_API_BASE_URL=http://localhost:8000`). This file is git-ignored so you don't leak secrets. Variables here MUST start with `VITE_` to be exposed to the React frontend code.
- **`.env.example`**: A safe template committed to version control. It shows new developers exactly which environment variables they need to configure locally.
- **`.github/workflows/ci.yml`**: Contains GitHub Actions workflows. This is an automated robot that runs every time you `push` or make a `pull request`. It installs dependencies, runs the TypeScript compiler, and runs unit tests to ensure no broken code enters the main branch.
- **`.storybook/main.ts`**: The core configuration for Storybook. It tells Storybook where to find your `.stories.tsx` files and which addons to use.
- **`.storybook/preview.tsx`**: The visual configuration for Storybook. Here we import the global Tailwind CSS (`index.css`) so that components rendered in Storybook look exactly like they do in the real app.
- **`.gitignore`**: Tells Git which files/folders to exclude from version control (like `node_modules`, `dist`, and `.env`).
- **`index.html`**: The main entry point for the browser. It contains the `<div id="root"></div>` where the React app is mounted, and it imports the `src/app/main.tsx` script to kickstart the JavaScript.
- **`package.json`**: The manifest file for the Node.js project. It defines the project name, executable scripts (like `npm run dev`, `npm run build`, `npm run storybook`), and lists exactly which dependencies (React, Zustand, Tailwind) and devDependencies (Vite, TypeScript, Testing Library) are installed.
- **`package-lock.json`**: Automatically generated by npm. It locks down the exact version tree of every dependency (including dependencies of dependencies) to ensure identical installations across all developer machines and CI pipelines.
- **`tsconfig.json`**: The main TypeScript configuration. It references other tsconfig files to keep concerns separated between Node (Vite config) and the browser (React app).
- **`tsconfig.app.json`**: The TypeScript compiler configuration specifically for your React code. It enforces strict typing rules, configures JSX handling, and sets up module resolution path aliases (e.g., mapping `@/` to the `src/` directory).
- **`tsconfig.node.json`**: TypeScript configuration specifically for files that run in Node.js, like `vite.config.ts`.
- **`vite.config.ts`**: Configuration for Vite, our modern build tool and dev server. It registers plugins (React, Tailwind CSS v4), configures the local development port, and sets up the `@/` path alias logic at the bundler level so Vite knows how to read your imports.
- **`vitest.config.ts`**: Configuration for Vitest (our testing framework). It merges with the Vite config but adds test-specific setups like using `jsdom` to simulate a fake browser environment in the terminal during tests.
- **`ARCHITECTURE.md`**: The technical manifesto for developers. It explains the high-level FSD rules, dependency flow constraints, and step-by-step guides for adding new features.

## 2. Source Directory (`src/`)

This is where all the application source code lives.

- **`src/env.d.ts`**: TypeScript declaration file. It augments Vite's `ImportMetaEnv` interface to provide strict typing and autocomplete for our custom environment variables. If you type `import.meta.env.VITE_`, TypeScript will suggest the correct variables.
- **`src/vite-env.d.ts`**: Standard Vite types for things like importing images, CSS modules, and SVGs into React files.

### 🏢 `src/app/` (The Application Wiring Layer)

This layer acts as the application's bootstrap. It wires together providers, routing, and global styles. **Crucially, it contains zero business logic.**

- **`main.tsx`**: The absolute execution entry point. It calls `createRoot().render(...)` to mount the React tree. It also dynamically bootstraps the MSW (Mock Service Worker) if mock mode is enabled before starting React.
- **`App.tsx`**: The root React component. It acts as a shell, composing the `AppProviders` (which hold the state contexts), the `RouterProvider` (which handles URLs), and global UI layers like the hot-toast notification system.
- **`styles/`**:
  - **`index.css`**: The global stylesheet. Instead of writing custom CSS, this file imports Tailwind CSS v4 directives. We also define the `dark` variant here so that Tailwind knows to trigger dark mode when the `.dark` class is applied to the HTML tag.
  - **`fonts.css`**: (If applicable) Used for `@font-face` declarations to load custom fonts.
- **`providers/`**:
  - **`AppProviders.tsx`**: A composer component. Instead of deeply nesting 10 different providers inside `App.tsx` and creating a "wrapper hell", they are all cleanly composed here.
  - **`ErrorBoundaryProvider.tsx`**: A global safety net. If a crash occurs anywhere in the app that isn't caught by a feature-level boundary, this prevents a "white screen of death" and renders the generic `ErrorFallback` UI instead.
  - **`QueryProvider.tsx`**: Initializes the React Query `QueryClient` and wraps the app in the `<QueryClientProvider>`. It handles caching, retries, and background refetching for API calls. It also injects the React Query Devtools for debugging.
  - **`ThemeProvider.tsx`**: Configures the `next-themes` library. It listens to the user's system preferences or local storage, and dynamically adds or removes the `.dark` class from the root HTML element, enabling Dark Mode across the entire app.
  - **`index.ts`**: The barrel export. Allows `App.tsx` to simply `import { AppProviders } from "./providers"`.
- **`router/`**:
  - **`routes.tsx`**: The master routing configuration using React Router v7's `createBrowserRouter`. It maps URL paths (like `/login`) to Layout wrappers and lazy-loaded page components.
  - **`LazyImports.ts`**: Centralizes all `React.lazy()` calls. By dynamically importing pages, Vite can split the JavaScript bundle into smaller chunks. This ensures users only download the code for the page they are currently viewing, making the app load instantly.
  - **`ProtectedRoute.tsx`**: A route guard wrapper. It checks if the user has an authentication token; if absent, it instantly redirects the user to `/login` before rendering any protected child routes.
  - **`PublicRoute.tsx`**: The inverse route guard. If an authenticated user (who is already logged in) tries to visit the `/login` or `/register` pages, it bounces them to the `/` home route.
  - **`index.ts`**: Barrel export.

### 🍅 `src/shared/` (The Generic Foundation Layer)

This layer contains globally reusable code that knows **nothing** about specific business features. If code mentions "User", "Example", "Order", or "Product", it does NOT belong here. 

- **`config/`**:
  - **`env.ts`**: Startup validation. It uses the `zod` library to parse `import.meta.env`. If required variables are missing or malformed, the app crashes immediately with a clear error in the console, rather than failing silently deep inside the app.
  - **`constants.ts`**: A centralized dictionary for "magic strings/numbers" (like default pagination sizes `PAGE_SIZE = 20`, local storage keys, or regex patterns).
  - **`routes.ts`**: Defines a `ROUTES` dictionary. We use `ROUTES.HOME` instead of typing `"/"` directly across the app. This prevents broken links if we ever decide to change a URL path in the future.
  - **`index.ts`**: Barrel export.
- **`lib/`**:
  - **`axios.ts`**: The globally configured Axios instance (`apiClient`). It sets the base URL. It also has an `interceptor` that runs before every outgoing request to inject the Authorization token. Another `interceptor` catches incoming responses globally, and if it sees a `401 Unauthorized` error, it automatically logs the user out and redirects to login.
  - **`queryClient.ts`**: A factory function that creates a `QueryClient` with our chosen production defaults (e.g., 5-minute stale times, disabling refetch-on-window-focus). Separated so it can be reused in test setups.
  - **`index.ts`**: Barrel export.
- **`types/`**:
  - **`api.types.ts`**: Global TypeScript interfaces defining the shape of our API responses (e.g., `ApiResponse<T>`, `PaginatedResponse<T>`).
  - **`common.types.ts`**: Reusable generic types like `ID` or `BaseEntity` (which includes standard fields like `id`, `createdAt`, `updatedAt`).
  - **`index.ts`**: Barrel export.
- **`utils/`**:
  - **`cn.ts`**: A crucial UI utility. It combines `clsx` (for writing conditional class logic cleanly) and `tailwind-merge` (to resolve conflicting Tailwind utility classes securely, e.g., merging `p-4` and `p-2` safely).
  - **`formatters.ts`**: Pure functions for transforming data for display (e.g., taking a raw ISO date and formatting it to "Jan 12, 2024", or formatting numbers into currency).
  - **`index.ts`**: Barrel export.
- **`hooks/`**:
  - **`useDebounce.ts`**: A generic custom hook that delays the update of a value. Essential for performance in live-search inputs so we don't spam the backend server with API requests on every single keystroke.
  - **`index.ts`**: Barrel export.
- **`ui/`** _(Dumb/Presentation Components)_:
  - **`Button/Button.tsx`**: A highly reusable button component supporting variants (primary, ghost, danger), sizes (sm, lg, icon), and loading states (disabling the button and showing a spinner).
  - **`Button/Button.stories.tsx`**: Storybook documentation file. It defines the different "Stories" (Primary, Ghost, Loading) so developers can visually test the Button outside of the application.
  - **`Spinner/Spinner.tsx`**: The generic loading indicator. Can be used inside buttons or rendered full-screen for page transitions.
  - **`Input/Input.tsx`**: A reusable input field for forms.
  - **`Modal/Modal.tsx`**: A generic overlay modal for dialogs and popups.
  - **`Card/Card.tsx`**: A surface container component for displaying grouped information.
  - **`ErrorFallback/ErrorFallback.tsx`**: The generic UI shown when a React Error Boundary catches a crash. It displays the error message and a "Try Again" button.
  - **`ThemeToggle/ThemeToggle.tsx`**: A toggle button utilizing Lucide icons to switch the app's `next-themes` state between `light` and `dark` modes visually.
  - **`index.ts`**: Barrel export making these primitives accessible via a clean import: `import { Button, ThemeToggle } from "@/shared/ui"`.
- **`layouts/`**:
  - **`AuthLayout.tsx`**: A structural shell for unauthenticated pages (like Login/Register). It usually centers content on the screen with a nice background. It uses `<Outlet />` to render the child pages.
  - **`MainLayout.tsx`**: The structural shell for the authenticated application. It contains the global header (which houses the `ThemeToggle`), a potential sidebar, and defines the main content area layout.
  - **`index.ts`**: Barrel export.

### 🍕 `src/features/` (The Business Logic Layer)

Features are isolated, vertical slices of business logic. They act as independent micro-applications. A feature owns everything related to its domain. We use `example-feature` as the blueprint:

- **`example-feature/api/exampleApi.ts`**:
  - Defines `exampleKeys`: A dictionary of React Query cache key factories. This ensures cache keys (like `["examples", "list"]`) are strictly typed and never misspelled across the app.
  - Defines the Axios calls (`getAll`, `getById`, `create`) specifically for this domain, using the `apiClient` from `shared/lib`.
- **`example-feature/api/exampleApi.test.ts`**:
  - Unit tests for the raw API functions, verifying correct endpoints and data mapping.
- **`example-feature/types/example.types.ts`**:
  - Defines the domain model interface (e.g., `interface Example { id: string, name: string }`).
  - Defines DTOs (Data Transfer Objects) for what the backend expects when creating/updating.
  - Defines Zod validation schemas to validate forms or incoming API responses securely.
- **`example-feature/hooks/`**:
  - **`useExamples.ts` & `useExample.ts`**: Wrappers around `useQuery` for fetching lists or single items.
  - **`useCreateExample.ts`, `useUpdateExample.ts`, `useDeleteExample.ts`**: Wrappers around `useMutation` that handle sending data to the server and invalidating cache on success.
  - **`useExampleFilters.ts`**: A UI-only hook for managing complex filtering logic locally before sending it to the API.
  - **`__tests__/*.test.ts`**: Unit tests verifying that the hooks fetch data correctly and update cache state.
- **`example-feature/store/exampleStore.ts`**:
  - A Zustand store. Used **only** for ephemeral Client State (e.g., which items are currently selected by checkboxes, or if a mobile filter panel is open). It never stores data fetched from the API (that is React Query's job).
- **`example-feature/store/exampleStore.test.ts`**:
  - Unit tests for the Zustand store to ensure local state actions mutate the state tree correctly.
- **`example-feature/components/`**:
  - **`ExampleCard.tsx`**: A domain-specific UI component rendering an `Example` object.
  - **`ExampleList.tsx`**: A "smart" component calling the `useExamples` hook and rendering a list of cards.
  - **`ExampleForm.tsx`**: A specific form for creating or editing examples, wiring up to `useCreateExample` or `useUpdateExample`.
  - **`ExampleFilters.tsx`**: A UI component that connects to `useExampleFilters` to allow users to filter the list.
  - **`ExampleErrorBoundary.tsx`**: A localized error boundary. If the `ExampleList` crashes, only this specific section shows an error, leaving the rest of the application functional.
  - **`__tests__/*.test.tsx`**: Component tests verifying UI rendering and interactions like button clicks.
- **`example-feature/utils/`**:
  - **`exampleTransformers.ts`**: Pure functions that map backend API schemas into frontend view models.
  - **`exampleValidation.ts`**: Pure functions for business logic validation on the client side.
  - **`__tests__/*.test.ts`**: Unit tests verifying the purity and logic of the utility functions.
- **`example-feature/index.ts`**:
  - **The Barrel Export (The Contract).** This is the most important file in the feature. It explicitly defines what other parts of the app are allowed to import from this feature. If a component is not exported here, it is considered private to the feature and cannot be used anywhere else.

### 📋 `src/pages/` (The Composition Layer)

Pages are incredibly "dumb". They contain almost zero logic. Their sole responsibility is to assemble components from features and shared layouts, then map them to a specific URL route.

- **`home/HomePage.tsx`**: The landing page composition. It renders static text, shared UI buttons, and routing links.
- **`dashboard/DashboardPage.tsx`**: Maps to the `/dashboard` URL. A protected page composition that can display analytics or user widgets.
- **`example-feature/ExampleListPage.tsx`**: Maps to the `/examples` URL. It renders a page header, a "Create" button, and embeds the `<ExampleList />` feature component inside the `<ExampleErrorBoundary />`.
- **`example-feature/ExampleDetailPage.tsx`**: Maps to `/examples/:id`. It uses `useParams` from React Router to read the URL parameter (the ID) and passes that ID down to a feature component to render details for that specific item.

---

### 🧩 `src/widgets/` (Composite UI Blocks)

Widgets are cross-feature UI blocks that combine shared UI elements and potentially pull data from multiple independent domains.

- **`header/Header.tsx`**: The global application header. Designed to compose features like an Auth profile dropdown and a notification bell.
- **`sidebar/Sidebar.tsx`**: The global navigation sidebar, rendering navigation links based on application routes.
- **`data-table/DataTable.tsx`**: A composite table structure that can be reused across different features to display tabular data uniformly.
- **`data-table/DataTableToolbar.tsx`**: A toolbar associated with the data table to provide filtering, search, and bulk actions.

---

### 🧪 `src/test/` (Testing Infrastructure)

Code here helps us simulate a fake backend and test the application without needing a real server.

- **`setup.ts`**: Runs automatically before our Vitest tests. It extends the testing environment with `jest-dom` matchers (like `.toBeInTheDocument()`) and starts the Mock Service Worker server to intercept network requests during tests.
- **`mocks/handlers.ts`**: Defines MSW network interceptors. Instead of hitting a real backend, when the app requests `/examples`, this file intercepts the call and returns fake JSON data instantly. This allows testing frontend logic without a real backend.
- **`mocks/browser.ts`**: Sets up MSW for the actual browser. If you start the app with `VITE_ENABLE_MOCKS=true`, you can develop the frontend completely isolated without needing a backend server running at all.
- **`mocks/server.ts`**: Sets up MSW for a Node.js environment, used strictly when running automated tests via the terminal.
- **`utils/renderWithProviders.tsx`**: A custom test utility that wraps components in all necessary global providers (like `QueryClientProvider`) during unit tests, ensuring components don't crash when testing React Query hooks.
- **`utils/testQueryClient.ts`**: An isolated React Query client specifically configured for tests (e.g., with retries disabled to prevent slow test execution).
